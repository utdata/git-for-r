18  Working with feature branches

üéØ Goal: By the end of this chapter, you‚Äôll know what a feature branch is, how to make one, and how to merge one back into your main project.

In Git, a branch is like a parallel universe for your code. You can work on a new idea or fix a bug without messing up the main project. When you‚Äôre ready, you bring your changes back to the main project with a merge.

18.1 Why Use Branches?

Let‚Äôs say you‚Äôre working with teammates. One of you is writing the introduction, another is cleaning the data, and a third is building plots.

If you all worked on the main branch at the same time, you‚Äôd likely constantly bump into each other ‚Äî accidentally overwriting work or creating conflicts. Branches let everyone work on their own part of the project safely.

18.2 üå± Step 1: Create a New Branch

Before we start, let‚Äôs look at what branches exist in our project.

  1. In your terminal, run the following command:

    git branch

You‚Äôll likely just see the main branch with an * symbol next to it. The asterisk tells you this is the branch you are currently working on in Git. We set this up at the start of our Git workflow.

Now, let‚Äôs say you wanted to try a new chart using the ggplot library, but you weren‚Äôt sure if you‚Äôre ready to use this in the full project. You‚Äôd like some room to experiment a bit before fully committing your work to the main project. This is where a feature branch comes in.

Before we jump into making our first branch, let‚Äôs cover some best practices when naming feature branches:

  • Use lowercase and hyphens (avoid spaces, underscores, or uppercase letters).
  • Keep names short but descriptive (include enough context to understand purpose: fix-overflow-navbar, not fix-bug).
  • Use consistent tense (preferably use a verb-noun format like add-navbar or fix-login-bug).

There are more conventions you would want to follow (such as using prefixes for branch types such as feature/, bugfix/, etc.) especially if working within a developer team, but for now these will be enough to get you started.

Let‚Äôs try making a new branch now.

  1. Run the following command in the terminal:

    git checkout -b add-plot-experiment

You should see a response in the terminal saying something like:

Switched to a new branch ‚Äòadd-plot-experiment‚Äô

This command did two things:

  1. checkout: Tells Git you want to switch to another branch.

  2. -b add-plot-experiment: the -b flag tells Git that you‚Äôll be creating a new branch called add-plot-experiment. Without the flag, Git would try to switch to a branch that already existed.

Now let‚Äôs confirm that you‚Äôre on the new branch.

  1. Run the branch command again:
git branch

You‚Äôll see something like:

*add-plot-experiment
main

Remember, the * shows your current branch. So you‚Äôre now on add-plot-experiment, a ‚Äúsandbox‚Äù that‚Äôs separate from your main branch in Git.

18.3 Step 2: make and commit changes on the new branch

Now that you‚Äôre on this new feature branch (which is an exact clone of our main branch at the time it is created), we can begin to safely experiment. You can make changes to our branch without worrying about messing up the work on our main branch.

Let‚Äôs try adding some new files to test this out.

In RStudio, do the following:

  1. Make a new quarto file called 01-plot.qmd

  2. Inside the file, make a new code block cell and add the following code:

    library(tidyverse)

ggplot(mtcars, aes(x=hp, y=mpg, color=cyl)) +
    geom_point(size=3)

  3. Run the cell to generate a simple ggplot() scatterplot.

  4. Save the file.

Now, we‚Äôll add the new plot file to Git by completing the normal Git workflow steps:

  1. Run git status to see what‚Äôs changed.
  2. Type and run git add 01-plot.qmd to tell Git to start tracking your new file.
  3. Run git status again to make sure your new file showing under ‚Äúchanges to be committed‚Äù.
  4. Run git commit -m "Add experimental plot" to take a snapshot of your changes.

This process is exactly the same as it has been for our other updates. However, this time you‚Äôve saved the snapshot of your file only to the add-plot-experiment branch, not to main.

Importantly, these changes have not been pushed to Github yet. We‚Äôll do that next.

18.4 Step 3: push the new branch to GitHub

Once you‚Äôve committed some changes on your branch, you‚Äôll probably want to back up your work by pushing it to Github.

The syntax to add a new branch to your repo is just like when we added our main branch during our initial project setup:

git push -u origin branch-name

This command does two things:

  1. Creates the new branch on the GitHub project.
  2. Links your local branch to it.

So if you wanted to add your new add-plot-experiment branch, for example, you‚Äôd type and run:

git push -u origin add-plot-experiment

Terminal would show a response like:

To github.com:username/my-git-quarto-project.git [new branch] add-plot-experiment -> add-plot-experiment branch ‚Äòadd-plot-experiment‚Äô set up to track ‚Äòorigin/add-plot-experiment‚Äô.*

If you go to your Github repo page, you‚Äôll now see the branch listed in your GitHub repo.

18.5 Merging back into main

If you‚Äôre happy with your changes on your feature branch and want to add it to your main project, you‚Äôll do so by merging the changes from your feature branch back into your main branch to incorporate them into your project.

To do so, follow the steps below.

Step 1: switch back to main

  1. Run the following to get back to your main branch:

    git checkout main

  2. Run git branch again. The asterisk should show next to main again.

Ok, now that we‚Äôre on a main branch, we want to bring in the changes from add-plot-experiment.

Heads-up: if you‚Äôve still got your 01-plot.qmd file open in R Studio when you switch back to your main branch, you‚Äôll get a pop-up telling you that the plot file has been deleted (see screenshot). That‚Äôs because the file doesn‚Äôt exist yet on the main branch. Hit the blue Yes button in the pop to close the file. Don‚Äôt worry though, the file still exists on your add-plot-experiment branch because you made a snapshot of your work with your git commit.ü§†

Step 2: merge the feature branch into main

  1. Type and run:

    git merge add-plot-experiment

This pulls the changes from your feature branch into the main project.

If everything goes smoothly, you‚Äôll see a message like:

Updating abc1234..def5678 Fast-forward myfile.qmd | 10 ++++++++++ 1 file changed, 10 insertions(+)

Note: If Git can‚Äôt automatically combine changes, you‚Äôll get a merge conflict. If this happens, you‚Äôll need to resolve the issues following the steps in the last chapter (see: How to resolve a merge conflict.

In your R Studio file pane, you should now see the 01-plot.qmd file listed (you may need to refresh).
